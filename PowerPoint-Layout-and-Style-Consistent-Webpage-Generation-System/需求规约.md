# 软件需求规约 (Software Requirements Specification)

**项目名称**：PowerPoint Layout-and-Style-Consistent Webpage Generation System

**版本**：1.0

**日期**：2026-04-12

---

## 1. 引言

### 1.1 目的

本文档旨在详细定义SEEM系统的功能性和非功能性需求，作为开发团队、测试团队以及所有利益相关者之间的共识基础。本需求规约遵循IRACIS原则，聚焦于提升营收（Increase Revenue）、避免成本（Avoid Costs）和改进服务（Improve Service）三个维度。

### 1.2 范围

本系统是一个基于"模板约束 + LLM生成"混合方法的智能PPT生成系统，将PPT页面抽象为"版式 + 组件 + 视觉风格"的可组合体系，最终输出HTML或Markdown格式的网页展示文件。

### 1.3 定义、缩略语和缩写

| 术语 | 定义 |
|------|------|
| 版式原子库 | 预设的最小布局单元集合，每个原子代表一种基本的页面结构模式 |
| 组件规格 | 标准化的UI组件定义，包括尺寸、间距、样式属性等元数据 |
| 风格令牌 | 控制视觉设计属性的变量集合，如颜色、字体、间距等 |
| 推理引擎 | 根据内容语义自动匹配最优版式的算法模块 |
| 元素重叠率 | 页面中元素边界相互覆盖的面积占总面积的比例 |
| 全局颜色偏差 | 多页之间颜色值的平均差异度 |

## 2. 总体描述

### 2.1 产品前景

在数字化办公场景中，传统的PPT制作面临三大痛点：手动排版耗时、多页风格难以统一、跨平台分发受限。本系统通过AI技术实现版式自动选型与风格化生成，使普通用户能在5分钟内完成一套专业级演示文稿的制作，同时保证跨平台兼容性和可编辑性。

### 2.2 产品定位

面向需要快速生成高质量演示文稿的企业员工、教师、学生等群体，提供"输入内容 → 选择风格 → 一键生成"的极简工作流。

### 2.3 用户特征

| 用户类型 | 特征描述 | 典型场景 |
|----------|----------|----------|
| 企业员工 | 追求效率，需快速产出汇报材料 | 周报、月报、项目汇报 |
| 教师 | 注重内容结构，需要图文混排 | 课件、教案、学术分享 |
| 学生 | 预算有限，追求个性化展示 | 答辩、演讲、社团活动 |
| 自由职业者 | 多平台分发需求强 | 客户提案、个人品牌展示 |

### 2.4 约束

- 输出格式：仅支持HTML和Markdown，不支持.pptx
- 技术基础：必须基于LLM代码生成能力
- 性能要求：元素重叠率必须为0，全局颜色偏差必须≤5%

## 3.2 功能需求详述

#### 3.2.1 内容输入模块

**功能描述**：接收用户输入的页面内容信息，包括标题、正文、要点列表、图片引用等，并将其结构化处理后传递给下游模块。

| 需求ID | 需求描述 | 优先级 | 用户故事 |
|--------|----------|--------|----------|
| FR-001 | 系统应支持文本内容的多级标题结构（1-4级） | P0 | 作为用户，我可以在输入框中键入多级标题，系统自动识别层级关系 |
| FR-002 | 系统应支持要点列表输入，每个要点包含标题和描述文本 | P0 | 作为用户，我可以输入多个要点，每个要点包含一个短标题和详细描述 |
| FR-003 | 系统应支持图片引用，通过URL或本地上传方式添加 | P1 | 作为用户，我可以为页面添加图片，图片将在生成时被放置在指定位置 |
| FR-004 | 系统应支持表格数据输入，包括表头和数据行 | P1 | 作为用户，我可以输入表格数据，系统将保留表格结构进行版式匹配 |
| FR-005 | 系统应支持图表元数据输入，标记数据类型和关键数据点 | P1 | 作为用户，我可以指定页面是否包含图表，系统据此选择适合的图表布局版式 |

**用户故事示例**：

> **US-001**: 作为市场部员工，我需要在准备季度汇报时上传或输入多页内容的文档，我希望系统能自动识别标题层级和要点结构，这样我只需专注于内容本身，而不必担心排版问题。

---

#### 3.2.2 版式自动选型模块（推理引擎）

**功能描述**：根据输入内容的语义特征（要点数量、文本长度、是否含图表等），通过约束求解或LLM推理自动匹配最适合的版式结构。

| 需求ID | 需求描述 | 优先级 | 用户故事 |
|--------|----------|--------|----------|
| FR-006 | 系统应基于要点数量自动选择版式类型（单要点/双要点/多要点布局） | P0 | 作为用户，当我输入3个要点时，系统自动选择"三列要点"版式而非"单列大图"版式 |
| FR-007 | 系统应根据文本长度自动调整版式的文字区域占比 | P0 | 作为用户，当我的文字内容较长时，系统会分配更多空间给文本区域，避免文字溢出，同时可以自动提取、筛选文本中的关键词 |
| FR-008 | 系统应在检测到图片时自动选择合适的图文并置版式 | P0 | 作为用户，当我的页面包含图片时，系统自动选择"左图右文"或"上图下文"等版式 |
| FR-009 | 系统应支持图表专用版式，根据图表数量选择单图表或双图表布局 | P0 | 作为用户，当我的页面需要展示数据图表时，系统选择带有图表区域的版式模板 |
| FR-010 | 系统应检测元素重叠风险，在生成前主动调整布局参数 | P0 | 作为用户，我期望生成的页面永远不会出现元素重叠，这是系统必须保证的底线 |
| FR-011 | 系统应支持版式预览，用户可在确认前查看多个版式候选 | P1 | 作为用户，我想在最终确定前预览系统推荐的多个版式方案，以便做出选择 |

**用户故事示例**：

> **US-002**: 作为项目经理，我需要在1小时内完成项目汇报演示文稿，包含文字、图表和图片。我希望系统能自动分析我上传的文档的内容特征，智能计算演示文稿的页数，并为每页匹配最合适的版式布局，无需我手动调整任何布局参数。

> **US-003**: 作为教师，我制作课件时经常遇到文字过多的页面。我希望系统能检测到内容密度，自动增加相关部分的页面或调整字体大小或调整文字区块的面积或提取筛选关键词来压缩文字篇幅，确保最终产出既美观又可读。

#### 3.2.3 风格令牌管理系统

**功能描述**：维护一套统一的风格令牌体系，确保多页生成时视觉风格的一致性。

| 需求ID | 需求描述 | 优先级 | 用户故事 |
|--------|----------|--------|----------|
| FR-012 | 系统应提供预设风格模板（如：商务蓝、学术灰、活力橙等）供用户选择 | P0 | 作为用户，我可以从多种预设风格中选择，系统将自动应用对应的颜色、字体和间距配置 |
| FR-013 | 系统应支持风格自定义，用户可调整主色、辅色、字体等参数 | P0 | 作为用户，我想自定义颜色主题，系统在用户调整后实时预览效果 |
| FR-014 | 系统应在多页生成时自动保持全局风格一致（颜色偏差≤5%） | P0 | 作为用户，当我生成多页PPT时，我期望每页的同级标题的颜色都完全一致，不出现色差，背景色也基本一致，略有差异 |
| FR-015 | 系统应支持风格令牌的导出和复用 | P1 | 作为用户，我可以将自己设计的风格保存为模板，下次使用时一键应用 |
| FR-016 | 系统应在风格调整时自动级联更新所有相关元素 | P0 | 作为用户，当我调整主色调时，系统应自动更新所有标题、按钮、背景等元素的颜色 |

**用户故事示例**：

> **US-004**: 作为品牌经理，我需要为客户生成一套品牌提案（共8页）。我希望能先选定公司的品牌色作为主色调，系统自动将这个颜色应用到所有8页的标题、强调色和图标上，确保整体视觉统一。

#### 3.2.4 LLM代码生成模块

**功能描述**：利用大语言模型的代码生成能力，结合版式结构、风格令牌和内容信息，生成符合规范的HTML/Markdown代码。

| 需求ID | 需求描述 | 优先级 | 用户故事 |
|--------|----------|--------|----------|
| FR-017 | 系统应生成符合W3C标准的HTML代码，确保跨浏览器兼容性 | P0 | 作为用户，我生成的HTML文件可以在Chrome、Firefox、Safari等浏览器中正常显示 |
| FR-018 | 系统应支持Markdown格式输出，适用于技术文档和博客平台 | P0 | 作为用户，我可以选择生成Markdown格式，直接粘贴到Notion或Typora中使用 |
| FR-019 | 系统应生成语义化的HTML结构，包含适当的标签和属性 | P0 | 作为用户，我生成的HTML代码结构清晰，便于后续SEO优化和二次编辑 |
| FR-020 | 系统应在代码生成时注入风格令牌，控制颜色、字体、间距等属性 | P0 | 作为用户，我选择的风格主题（如商务蓝）应体现在生成的每个元素的CSS属性中 |
| FR-021 | 系统应支持代码的语法检查，标记潜在的格式错误 | P1 | 作为用户，当LLM生成的代码存在格式问题时，系统应提示并允许我手动修复 |
| FR-022 | 系统应支持增量生成，用户修改某一页时只重新生成该页 | P1 | 作为用户，当我修改第3页的内容时，系统只需重新生成第3页，其他页面保持不变 |

**用户故事示例**：

> **US-005**: 作为技术博主，我需要将我的PPT内容转换为博客文章。我希望系统能直接生成Markdown格式，包含正确的标题层级、列表和图片链接，我可以一键复制到我的博客系统中。

## 6. 验收标准

### 6.1 功能验收标准

| 功能模块 | 验收通过条件 |
|----------|--------------|
| 内容输入 | 支持标题(1-4级)、要点列表、图片、表格的完整输入 |
| 版式选型 | 根据内容特征自动匹配正确版式，无人工干预 |
| 风格管理 | 风格选择和自定义功能完整，全局一致性达标 |
| 代码生成 | 生成符合W3C标准的HTML/Markdown代码 |
| 预览编辑 | 预览、编辑、导出功能正常运作 |
| 量化评估 | 元素重叠率和颜色偏差计算准确，报告可导出 |

### 6.2 性能验收标准

| 指标 | 目标值 | 测试方法 |
|------|--------|----------|
| 元素重叠率 | 0% | 自动布局测试集（100个样本）重叠率均值 |
| 全局颜色偏差 | ≤5% | 多页生成测试集（50套）颜色偏差均值 |
| 单页生成时间 | ≤10s | 计时工具测量100次取平均值 |
| 批量生成时间 | ≤60s | 计时工具测量10页套件生成时间 |
| 系统并发能力 | ≥5 | 5用户同时发起请求无明显延迟 |

### 6.3 成功标准（IRACIS）

| 维度 | 指标 | 目标 |
|------|------|------|
| Increase Revenue | 产出专业级演示材料，开辟新型演示分发价值 | 生成的网页支持嵌入企业官网、外部分享 |
| Avoid Costs | 视觉排版时间相比传统方法减少 | ≥50% |
| Improve Service | 高质量美观产出，专业布局建模与生成式AI融合 | 元素重叠率0%，颜色偏差≤5% |
